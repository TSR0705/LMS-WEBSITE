# Contributing to the LMS

Thank you for your interest in contributing! This document outlines the standards, workflows, and conventions we use to keep this codebase clean, secure, and maintainable.

---

## 1. Project Standards

This is a production repository containing sensitive payment and identity logic. 
- **Security First:** Never commit secrets. Ensure any new Server Actions strictly validate user identity via `auth()` before mutating the database.
- **Server-First Mindset:** Default to React Server Components. Only use `"use client"` when interactivity (hooks, state, event listeners) is strictly required. Keep client components as small as possible.
- **Type Safety:** We use TypeScript heavily. `any` types will cause the build to fail. Always define proper interfaces or use the autogenerated Sanity types (`sanity.types.ts`).

## 2. Branch Strategy

We follow a simplified Git Flow:

- `main`: The production branch. This branch is locked and automatically deploys to Vercel. Code here must always be stable.
- `feature/*`: Create a feature branch for new functionality (e.g., `feature/certificate-generation`).
- `fix/*`: Create a fix branch for bugs (e.g., `fix/video-hydration-error`).
- `chore/*`: For maintenance, dependency updates, and documentation (e.g., `chore/update-readme`).

## 3. Commit Conventions

We enforce [Conventional Commits](https://www.conventionalcommits.org/). This allows us to auto-generate changelogs and understand history at a glance.

Format: `<type>(<scope>): <subject>`

**Examples:**
- `feat(auth): add magic link sign-in option`
- `fix(stripe): correct webhook signature validation logic`
- `docs(readme): add environment variable section`
- `chore(deps): update next.js to 15.3.9`

*The `<scope>` is optional but highly recommended.*

## 4. Coding Style

- **Formatting:** We use Prettier. Ensure your editor is configured to format on save.
- **Linting:** `eslint` is strictly enforced. Run `npm run lint` before committing.
- **Tailwind:** Use utility classes directly in the `className` prop. If composing complex dynamic classes, use the `cn()` utility (`lib/utils.ts`) which merges classes using `clsx` and `tailwind-merge` to avoid specificity collisions.
- **Data Fetching:** Do not use `fetch` on the client side unless absolutely necessary. Fetch data in the Server Component using `sanityFetch` and pass it down as props.

## 5. Pull Request Checklist

Before opening a PR, ensure you can check off every item on this list:

- [ ] My branch is up to date with `main`.
- [ ] I have run `npm run build` locally and it succeeds with 0 errors.
- [ ] I have run `npm run lint` and resolved all warnings/errors.
- [ ] I have not exposed any new environment variables to the browser unintentionally.
- [ ] If my PR requires a new environment variable, I have added it to `.env.example` and documented it in `docs/ENVIRONMENT.md`.
- [ ] If my PR changes database structures, I have updated the Sanity schemas in `sanity/schemaTypes/`.

## 6. Testing Expectations

While this project currently relies heavily on Next.js type-checking and Vercel build checks, any modifications to the following critical paths require rigorous **manual testing** before approval:
1. **The Enrollment Gate:** `app/(dashboard)/courses/[courseId]/page.tsx`
2. **The Stripe Checkout Flow:** `actions/createStripeCheckout.ts`
3. **The Webhook Processor:** `app/api/stripe-checkout/webhook/route.ts`

Any regressions in these three files will directly impact revenue or security. Document your manual test results directly in your Pull Request description.
